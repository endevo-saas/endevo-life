## 1. Role Overview

One shared execution role is used by **all 5 Lambda functions**:

| Function | Role |
|----------|------|
| endevo-uat-fn-auth | endevo-uat-lambda-role |
| endevo-uat-fn-admin | endevo-uat-lambda-role |
| endevo-uat-fn-hr | endevo-uat-lambda-role |
| endevo-uat-fn-employee | endevo-uat-lambda-role |
| endevo-uat-fn-lms | endevo-uat-lambda-role |

**Finding: SHARED ROLE** -- All functions share one role, meaning every function has every permission. The auth function can delete S3 objects; the employee function can admin-create Cognito users. This violates least-privilege.

## 3. Wildcard Findings

### CRITICAL: Resource = `*` (Full Wildcard)

| # | Statement | Actions | Severity | Risk |
|---|-----------|---------|----------|------|
| W1 | SESAccess | ses:SendEmail, ses:SendRawEmail | **HIGH** | Lambda can send email from ANY verified identity in the account, not just endevo identities. An attacker with Lambda access could send phishing emails from any SES-verified domain. |
| W2 | CloudFrontSignedUrls | cloudfront:CreateInvalidation, cloudfront:GetDistribution | **MEDIUM** | Lambda can invalidate caches and read config of ANY CloudFront distribution in the account. Note: Policy 3 already has the correctly scoped version for the same action. |
| W3 | AWSLambdaBasicExecutionRole | logs:CreateLogGroup, CreateLogStream, PutLogEvents | **LOW** | AWS managed policy; industry-standard. Can write to any log group, but this is normal for Lambda. Acceptable risk, though scoping to `/aws/lambda/endevo-uat-*` log groups is ideal. |

### MEDIUM: Prefix Wildcards (endevo-uat-*)

| # | Statement | Pattern | Risk |
|---|-----------|---------|------|
| W4 | DynamoDBAccess | `table/endevo-uat-*` | Grants access to ALL current and future tables with this prefix, including tables the function may not need. Also grants `dynamodb:Scan` which is expensive and rarely needed in production. |
| W5 | S3Access / S3ListAccess | `endevo-uat-*` | Grants access to ALL current and future buckets with this prefix. Any new bucket matching this prefix is automatically accessible. |

## 4. Over-Broad Permissions

### 4a. Shared Role (Most Serious)

All 5 functions share one role. Actual need per function:

| Function | Actually Needs |
|----------|---------------|
| fn-auth | Cognito (admin ops), DynamoDB (users/tenants table), SES (verification emails) |
| fn-admin | DynamoDB (all tables), Cognito (admin user mgmt) |
| fn-hr | DynamoDB (employee/tenant tables), SES (invites), Cognito (create users) |
| fn-employee | DynamoDB (employee tables), S3 (document uploads) |
| fn-lms | DynamoDB (LMS tables), S3 (video content), CloudFront (invalidation) |

**Current state:** Every function has Cognito admin, S3 delete, SES send, CloudFront invalidation, and DynamoDB scan on all tables. The employee function should NOT be able to AdminDeleteUser in Cognito. The auth function should NOT be able to delete S3 objects.

### 4b. DynamoDB: Scan Permission

`dynamodb:Scan` is granted globally. Scans are expensive, can cause throttling, and are rarely needed. Only admin reporting functions should have it.

### 4c. DynamoDB: DeleteItem on All Tables

All functions can delete items from all tables. Only admin functions should have delete capability on most tables.

### 4d. CloudFront Duplicate + Wildcard

Policy 2 grants `cloudfront:CreateInvalidation` and `cloudfront:GetDistribution` on `*` (all distributions). Policy 3 correctly scopes `cloudfront:CreateInvalidation` to a specific distribution. Policy 2's wildcard makes Policy 3 redundant and is the one that should be removed/scoped.

## 5. Least-Privilege Recommendations

### Priority 1: Split into Per-Function Roles (HIGH)

Create 5 separate IAM roles:

```
endevo-uat-role-auth      -> Cognito + DynamoDB (users, tenants) + SES
endevo-uat-role-admin     -> DynamoDB (all tables) + Cognito + SES
endevo-uat-role-hr        -> DynamoDB (employees, tenants) + Cognito (create) + SES
endevo-uat-role-employee  -> DynamoDB (employees, documents) + S3
endevo-uat-role-lms       -> DynamoDB (lms tables) + S3 + CloudFront (specific dist)
```

### Priority 2: Scope SES to Specific Identities (HIGH)

Replace `Resource: *` with specific SES identity ARNs:
```
arn:aws:ses:us-east-1:383423735462:identity/noreply@endevolife.com
arn:aws:ses:us-east-1:383423735462:identity/endevolife.com
```

### Priority 3: Scope CloudFront to Specific Distribution (MEDIUM)

Replace `Resource: *` with:
```
arn:aws:cloudfront::383423735462:distribution/E121OSHNXKRE61
```
And remove Policy 2's CloudFrontSignedUrls statement (keep only Policy 3's scoped version, adding `GetDistribution` to it).

### Priority 4: Scope CloudWatch Logs (LOW)

Replace the managed `AWSLambdaBasicExecutionRole` with a custom policy:
```json
{
  "Effect": "Allow",
  "Action": ["logs:CreateLogGroup", "logs:CreateLogStream", "logs:PutLogEvents"],
  "Resource": "arn:aws:logs:us-east-1:383423735462:log-group:/aws/lambda/endevo-uat-*:*"
}
```

### Priority 5: Remove Scan Where Not Needed (MEDIUM)

Remove `dynamodb:Scan` from all roles except admin. Use Query with proper indexes instead.

### Priority 6: Scope DynamoDB to Specific Tables (MEDIUM)

Instead of `table/endevo-uat-*`, list explicit table ARNs per role:
- fn-auth: `endevo-uat-users`, `endevo-uat-tenants`
- fn-employee: `endevo-uat-employees`, `endevo-uat-documents`
- etc.
